# 大规模用户模拟系统设计文档

> 短视频推荐场景 | 闭环校准架构 | v0.1
>
> 日期: 2026-05-06

---

## 1. Context

### 1.1 问题

推荐系统的离线评估、压力测试和方法论研究都需要大规模、分布逼真的用户行为数据。真实数据受限于隐私、成本和覆盖面，无法按需生成特定条件下的用户行为。

### 1.2 目标

用 1000 个 AI agent 高保真模拟用户行为，通过统计外推覆盖 10 亿用户规模的分布，服务于三个场景:

| 场景 | 需求 |
|------|------|
| 离线算法评估 | 分布保真度高，能做可信的 A/B test |
| 压力测试 | 规模可扩展，能模拟 10 亿量级流量 |
| 方法论研究 | 可验证、可复现，推动 "LLM agent 模拟真实用户" 的学术边界 |

### 1.3 关键约束

- 1000 个 LLM agent 是成本上限（高保真层）
- 模拟场景为短视频推荐（类抖音/YouTube）
- 需要覆盖全部分布维度: 人口统计、兴趣、活跃度、行为模式、时序特征
- 交互层需要模拟基础设施变量（画质、卡顿、编码器、成本）和场景上下文（首刷、时段、网络、设备）

## 2. 系统架构

### 2.1 分层总览

```
┌─────────────────────────────────────────────────┐
│            评估层 (Evaluation)                    │
│  A/B test 框架 / 算法对比 / 指标看板               │
│  所有结论附带保真度系数 F                          │
├─────────────────────────────────────────────────┤
│         外推层 (Scale-up Engine)                  │
│  Vine Copula + OT 权重 → 10 亿虚拟用户行为流       │
│  追踪外推放大偏差 F_scale                         │
├─────────────────────────────────────────────────┤
│         校准层 (Calibration Loop)                 │
│  三嵌套循环 / 分布对比 / 参数回调 / 收敛判定        │
│  分层保真度: 边缘>0.95 / 二阶>0.85 / 高阶标注      │
├─────────────────────────────────────────────────┤
│  交互层 (Agent x RecSys x Infra x Context)       │
│  三层决策引擎 / 干预控制台                         │
│  f(用户画像, 内容+基础设施, 场景上下文)             │
├─────────────────────────────────────────────────┤
│         Persona 生成层                            │
│  运筹学优化布局 (正交设计+Support Points+列生成)    │
│  两阶段生成: 数学骨架 → LLM 血肉填充               │
├─────────────────────────────────────────────────┤
│         基座层 (Distribution Baseline)            │
│  KuaiRec(主) + MIND + MovieLens                  │
│  统一 schema → 聚类 → 联合分布建模                 │
└─────────────────────────────────────────────────┘
```

### 2.2 层间通信

层间通过**标准化的分布描述格式**通信（不是原始数据），每层可独立迭代。交互层是唯一调用 LLM 的地方，成本集中在此。外推层是纯数学/统计，不依赖 LLM，可高速运行。

## 3. 基座层: 真实分布提取

### 3.1 数据源

| 数据集 | 角色 | 路径 |
|--------|------|------|
| KuaiRec | 主数据源（短视频，最接近目标场景） | `/Users/bytedance/Desktop/hehe/datasets/KuaiRec/` |
| MovieLens-25M | 补充评分行为和长尾兴趣分布 | `/Users/bytedance/Desktop/hehe/datasets/ml-25m/` |
| MIND | 补充内容消费模式和时序行为 | `/Users/bytedance/Desktop/hehe/datasets/MIND/` |

### 3.2 处理流程

```
Step 1: 统一行为表示
  不同数据集 → 标准化 schema
  (user_id, item_id, action_type, duration, timestamp, context)

Step 2: 用户聚类
  多维特征 → 降维 → 聚类
  输出: 50-100 个 archetype
  每个 archetype 包含:
    - 人口统计中心
    - 兴趣向量（品类分布）
    - 活跃度参数（日均时长/频次）
    - 行为指纹（完播率/互动率分布）
    - 时序模式（活跃时段/兴趣漂移速率）

Step 3: 联合分布建模
  - archetype 间的比例分布
  - archetype 内各维度的条件分布
  - 维度间相关性矩阵
  - 存储格式: 可序列化的分布描述文件

Step 4: 数据集差异补偿
  加权融合，KuaiRec 为主
```

### 3.3 基座层输出

每份分布描述附带 `F_baseline`（数据集本身的置信度），标明该估计基于多少样本、哪些维度是直接观测的、哪些是推断的。

## 4. 保真度度量体系

贯穿整个系统的量化指标框架。

### 4.1 分层指标

```
保真度度量体系 (Fidelity Metrics)
│
├── 单维度偏差
│   ├── 兴趣分布偏差:  KL(P_real || P_sim)  per 品类
│   ├── 活跃度偏差:    Wasserstein(真实, 模拟)
│   ├── 行为率偏差:    |完播率_real - 完播率_sim| / 完播率_real
│   └── 时序偏差:      DTW(真实活跃时序曲线, 模拟时序曲线)
│
├── 联合分布偏差
│   ├── 维度间相关性偏差:  ||Corr_real - Corr_sim||_F
│   ├── 条件分布偏差:     KL(P(行为|条件)_real || P(行为|条件)_sim)
│   └── 聚类结构偏差:     ARI(真实聚类, 模拟数据重新聚类结果)
│
├── 外推放大偏差
│   ├── 1K→10K 放大后分布漂移量
│   ├── 10K→1亿 放大后分布漂移量
│   └── 放大系数 vs 偏差增长曲线（理想: 亚线性）
│
└── 综合保真度系数 F
    F = Sigma(wi * (1 - di / di_max))  in [0, 1]
    wi = 各维度权重（可配置）
    di = 各维度偏差值
    di_max = 该维度最大可接受偏差
```

### 4.2 保真度等级

| F 值 | 等级 | 适用场景 |
|------|------|----------|
| > 0.9 | 高保真 | 可用于算法评估、论文发表 |
| > 0.7 | 中保真 | 可用于趋势判断、压力测试 |
| < 0.7 | 低保真 | 需要重新校准 |

### 4.3 嵌入位置

```
基座层产出 → 附带 F_baseline（数据集本身的置信度）
交互层产出 → 附带 F_agent（agent 行为 vs 基座的偏差）
校准层判定 → F_agent < 阈值 → 触发重新校准
外推层产出 → 附带 F_scale（放大后的偏差增长率）
评估层展示 → 所有实验结论旁边都显示当前 F 值
```

## 5. Persona 生成层

### 5.1 核心问题: 1000 人能否代表真实分布

直接回答: 在高维联合分布上不够。维度灾难使得 43000+ 格子中绝大多数为空。

**分层放弃精度**:

| 层级 | 1000 人能做到 | 保真度目标 |
|------|--------------|-----------|
| 边缘分布 | 各维度独立对齐，样本量充足 | F > 0.95 |
| 二阶条件分布 | 选 20-30 个关键条件组合，每组 30+ agent | F > 0.85 |
| 高维长尾 | 靠 LLM 世界知识做合理猜测 | F_estimated，加 warning |

### 5.2 运筹学优化布局

不是"从分布里采样 1000 个人"，而是"在高维空间里最优放置 1000 个点"。

```
Phase A: 初始布局 — 正交设计 (200 个)
  用正交表 / Latin Hypercube Sampling
  保证所有主效应 + 关键二阶交互全覆盖
  数学保证: 可估计所有主效应

Phase B: 最优填充 — Support Points (500 个)
  用 Wasserstein 距离优化
  min W2(P_real, Sigma(wi * delta(xi)))
  补充联合分布的薄弱区域
  每个 agent 计算代表权重 wi

Phase C: 迭代补盲 — 列生成 (300 个)
  跑一轮交互 → 计算保真度 F
  找 F 最差的区域
  用对偶变量（shadow price）计算边际收益最大的位置
  在该区域生成新 agent
  重复直到 F 收敛或名额用完

Phase D: 外推加权 — 最优传输权重
  每个 agent 带权重 wi
  权重从 OT 求解得到，不是简单等比放大
```

### 5.3 两阶段 Persona 生成

**阶段 1: 骨架分配（数学驱动，不用 LLM）**

从 archetype 的条件分布中采样每个 agent 的具体参数:
- 年龄 ~ N(mu, sigma)
- 兴趣权重向量 ~ Dirichlet(alpha)
- 活跃度 ~ LogNormal(mu, sigma)
- 行为基线率 ~ Beta(a, b)
- 时序参数（活跃时段、疲劳衰减率）

**阶段 2: 血肉填充（LLM 驱动）**

LLM 将数值参数翻译成自然语言身份，同时生成:
- 性格特征（耐心度、尝新意愿、从众倾向）
- 决策风格（冲动型/理性型/社交驱动型）
- context 敏感度（对画质/延迟/时段的反应模式）
- 个人叙事（背景故事）

**骨架参数约束 LLM 决策**: 骨架说完播率基线 40%，LLM 在此约束内做 context-sensitive 决策——同样 40% 的人，感兴趣+好画质会完播，不感兴趣+卡顿会秒划，而不是机械地每 10 个看完 4 个。

## 6. 高保真交互层

### 6.1 三维输入

```
决策输入 = (用户画像, 内容+基础设施状态, 场景上下文)
```

**内容 + 基础设施**:

| 类别 | 变量 |
|------|------|
| 推荐引擎 | 算法选择、排序策略 |
| 内容池 | 视频库、标签、时长、类型 |
| 带宽/成本 | CDN 成本、用户网络环境 (2G/4G/WiFi) |
| 视频画质 | 360p / 720p / 1080p / 4K 自适应 |
| 编码器 | H.264 / H.265 / AV1、码率、关键帧间隔 |
| 加载性能 | 首帧时间、卡顿率、预加载量 |
| 端侧算力 | 机型分布影响解码能力 |

**场景上下文**:

| 类别 | 变量 |
|------|------|
| 会话状态 | 首刷 / 普通 / 回流 / 会话深度（疲劳曲线） |
| 时空环境 | 时段 / 地理 / 场景（碎片化 vs 沙发时间） |
| 设备状态 | 电量 / 屏幕尺寸 / 后台应用竞争 |
| 用户即时状态 | 情绪/意图 / 社交触发（分享进入 vs 自然打开） |

**干预控制台**: 实验者可调节任意变量，例如"把 30% 用户的画质从 1080p 降到 720p，观察留存变化"。

### 6.2 决策分层

三层按职责划分，不是按成本划分:

```
Layer 0: 体验决策层（最重要，要最准）
  "用户因为体验问题离开，跟内容无关"
  决定: 用户还愿不愿意继续用这个产品

Layer 1: 内容决策层（要符合大盘分布）
  "体验没问题的前提下，用户对内容的反应"
  决定: 用户喜不喜欢推荐的内容

Layer 2: 涌现决策层（处理复杂/交叉场景）
  "多因素交织时的不确定性决策"
  决定: 无法用前两层规则覆盖的场景
```

### 6.3 Layer 0: 体验决策层 — 精确建模

**体验-行为响应函数**:

| 体验因素 | 可衡量技术指标 | 用户行为映射 |
|----------|--------------|-------------|
| 卡顿 | 卡顿率(%) / 卡顿时长(ms) / 次数/分钟 | 划走概率 / 退出概率 / 次日留存 |
| 画质 | 分辨率 / 码率(kbps) / VMAF 分数 | 完播率变化 / 点赞率变化 / 分享意愿 |
| 首帧延迟 | 首帧时间(ms) | 首刷跳出率 / 等待→划走阈值 |
| 编码效率 | 同画质带宽占用 (H264/H265/AV1) | 用户无感，影响企业成本 |
| 加载策略 | 预加载量(秒) / 缓冲区大小 | 滑动切换流畅度 |

**不同用户群的响应曲线不同**:
- 一线城市 WiFi 用户: 容忍度低，曲线陡
- 三线城市 4G 用户: 容忍度高，曲线缓
- 首刷新用户: 容忍度最低，曲线最陡

**校准源**（优先级从高到低）:
1. 行业公开报告（Akamai/Conviva 等 CDN 厂商的 QoE 研究）
2. 学术论文（QoE 领域、ITU-T P.1203 等标准）
3. KuaiRec 数据中的间接信号
4. 企业自有 A/B test 历史数据

**Layer 0 与企业成本联动**:

模拟系统能回答: "把全量用户从 1080p 降到 720p → 带宽成本省 X%、完播率降 Y%、留存降 Z%、ROI = 成本节省 / 留存损失"。

**保真度要求**:
- 体验-行为响应函数 MAPE < 10%
- 首刷场景单独校准 MAPE < 5%
- 偏差方向的后果: 高估容忍度→过度降质→用户流失；低估容忍度→过度投入→成本浪费

### 6.4 Layer 1: 内容决策层 — 对齐大盘

目标不是"每个人的决策要对"，而是"1000 人整体的内容消费分布要对"。

需要对齐的大盘指标:
- 品类消费占比分布
- 平均完播率 per 品类
- 互动率（点赞/评论/分享）per 品类
- 视频时长偏好分布
- 关注转化率

实现: 参数化模型驱动，定期用 LLM 抽检校验。

### 6.5 Layer 2: 涌现决策层 — LLM 精细决策

触发条件:
- 首刷前 5 个视频（留存关键路径）
- 新品类首次遭遇（兴趣探索）
- 多因素冲突（内容好但画质差、不感兴趣但朋友分享）
- 校准层要求特定区域需要高保真数据
- 随机抽样 10% 的普通决策做 LLM 校验

### 6.6 三层联动示例

```
推荐系统选出视频 V
    |
    v
Layer 0: 首帧延迟 1.8s + 4G 网络
  → 体验-行为函数 → P(等待)=0.6 → 采样 → 决定等待
  → 播放中卡顿 2 次，画质 480p → watch_pct 打折系数 = 0.7
    |
    v
Layer 1: 内容匹配度 = 兴趣向量 · 内容向量
  → 基础完播概率 = 0.65
  → 被 Layer 0 打折 → 0.65 x 0.7 = 0.455
  → 采样 → 看了 45% 划走
    |
    v
最终行为: watch_pct=0.45, no_like, no_comment
标注: layer0_factor=0.7, layer1_base=0.65

可拆解出:
  "如果画质正常, 完播率是 0.65"
  "画质差导致完播率从 0.65 降到 0.455"
  "体验损失 = 30%"
```

### 6.7 成本估算

```
1000 agents x 30 视频/会话 = 30,000 次决策/天

Layer 0 规则短路:   ~10% =  3,000 次 → $0
Layer 1 参数化:     ~70% = 21,000 次 → $0
Layer 2 LLM:        ~20% =  6,000 次 → ~3M tokens/天 ≈ $15-45/天
```

### 6.8 LLM 调用优化

- **批量共享**: 同 archetype 的 agent 遇到相似决策时共享 LLM 调用结果 + 个体扰动
- **缓存机制**: 相似 (archetype x content_category x infra_bucket x context_bucket) 的决策可缓存
- **Teacher-Student**: Layer 2 LLM 做 teacher，Layer 1 参数化模型做 student，校验轮次增加后 LLM 调用比例递减

### 6.9 行为日志 Schema

```json
{
  "session_id": "...",
  "agent_id": "...",
  "step_index": 5,
  "video_features": {},
  "infra_state": {"bitrate": 2400, "codec": "h265", "latency_ms": 180, "quality": "720p"},
  "context": {"session_type": "normal", "time_slot": "evening", "network": "4g", "fatigue": 0.3},
  "decision_layer": 1,
  "action": "watch",
  "watch_pct": 0.45,
  "interactions": {"liked": false, "commented": false, "shared": false},
  "reason": null,
  "interest_drift": {"food": 0.01, "tech": -0.005},

```

`fidelity_tag` 取值: `rule` | `parametric` | `llm` | `cached`。下游分析可按标签过滤，选择信度和量的 tradeoff。

### 6.10 推荐系统接口

```
interface RecSys:
  input:  user_profile, history, context
  output: ranked_list[{
    video_id, content_features,
    infra_state,
    rank_score
  }]
```

可插入不同算法做 A/B 对比、插入降级策略（成本压力下降画质）、插入 baseline（随机推荐）做对照组。

## 7. 校准层

### 7.1 三嵌套循环

```
外循环: Persona 重构 (代价最高，尽量少触发)
  触发条件: 边缘分布偏差 > 15%
  动作: 回到 Persona 生成层，用运筹学方法重新布局
  频率: 理想状态下只在初始化阶段跑 2-3 轮

  中循环: 参数微调 (中等代价)
    触发条件: 条件分布偏差 > 10%
    动作: 调整 persona 骨架参数
      - 兴趣权重向量微调
      - 行为基线率校正
      - Layer 0 体验响应曲线参数修正
    频率: 每轮模拟后检查一次

    内循环: Layer 1 <-> 2 校验 (低代价)
      触发条件: 持续运行
      动作: Layer 2 LLM 抽检 Layer 1 参数化决策的准确性
      如果偏差大 → 更新 Layer 1 参数
      频率: 每 N 条决策抽检一次
```

### 7.2 校准指标对比矩阵

校准时产出一张对照表，覆盖边缘分布、条件分布、体验响应、时序特征等所有维度，每个维度标注真实值、模拟值、偏差值和是否达标。

### 7.3 自动修正逻辑

```
找到偏差 → 归因（属于哪层/哪个参数）
  → 定位参数 → MLE 重拟合或梯度小步调整
  → 验证（目标改善 + 其他指标不被破坏）
```

### 7.4 防止过拟合

1. **留出验证集**: KuaiRec 按时间 70/30 切分，校准集 vs 验证集的 F 差异监控 overfit
2. **交叉数据集验证**: 在 KuaiRec 上校准的参数拿到 MovieLens 上检验泛化性
3. **参数正则化**: |theta_calibrated - theta_prior| < lambda * sigma_prior，防止偏离 prior 太远

## 8. 外推层

### 8.1 核心原则

外推不是"1000 人的行为 x 100 万倍"（只是重复同样的模式），而是"从 1000 人的行为中提取生成模型，采样出 10 亿个各不相同的个体"。

### 8.2 三阶段流程

**Stage 1: 拟合生成模型**

从 1000 agent 的行为日志中提取特征向量，拟合联合分布的生成模型。

| 候选模型 | 优点 | 风险 |
|----------|------|------|
| 多维 GMM | 简单可解释，拟合快 | 高维下组件数爆炸 |
| Vine Copula | 边缘分布和相关性分离，灵活 | 尾部依赖建模需要经验 |
| Normalizing Flow | 任意复杂分布都能拟合 | 1000 样本可能不够训练 |
| VAE | 有隐空间可插值 | 同上 |

**推荐: Vine Copula** — 边缘分布用非参数 KDE，相关性用 vine copula 建模，1000 个样本足够拟合 vine 结构。

**Stage 2: 带权采样**

- N=10,000: 直接采样，检查是否在训练数据覆盖范围内
- N=10 亿: 生成 10,000-50,000 个代表性个体，每个附带权重 wi，总和=10 亿。实际行为由 Layer 1 参数化模型驱动，不再调 LLM。

**Stage 3: 外推质量验证**

1. **自洽性检验**: 从 10 亿虚拟用户中随机抽 1000 个，行为分布应 ≈ 原始 1000 agent
2. **放大稳定性**: 生成 1K/10K/100K/1M/10 亿，指标应随 N 增大趋于稳定
3. **联合分布保持**: ||Corr(1K) - Corr(10 亿)||_F < epsilon

### 8.3 输出格式

外推层不输出 10 亿条明细数据，而是输出一个 **TrafficGenerator**:

```
TrafficGenerator:
  input:  time_window, infra_config, rec_algorithm
  output: {
    total_dau,
    per_archetype_metrics: {
      user_count, avg_session_len,
      avg_completion_rate, avg_like_rate,
      churn_rate_at_this_infra
    },
    infra_impact: {
      "场景描述": {
        completion_rate_delta,
        dau_delta,
        bandwidth_saving,
        cost_saving_daily
      }
    },
    fidelity: {
      F_overall, F_marginal, F_conditional,
      F_extrapolation, warning
    }
  }
```
